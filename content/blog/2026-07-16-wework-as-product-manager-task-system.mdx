同时跑三个项目，是一种什么感觉？

异乡人才、异乡点评、dingning.ai——三条线并行，技术团队在推进开发，我在中间做决策、对齐优先级、跟进卡点。没有专职 PM，没有项目管理软件，只有一个随时弹出消息的企业微信。

最开始，事情就是掉进缝隙里的。一个功能评审完了，但没人知道下一步是谁来执行；一个 PR 合并了，但我不知道上线了没有；一个任务在群里说好了，下周回来全员失忆。

不是团队的问题。是我没有把「决策节奏」变成「系统」。

---

## 一个非技术主理人的真实处境

我不写代码。准确说，我用 AI 辅助写代码，但我不是工程师——我是这几个项目的主理人，负责的是「这件事值不值得做」「做到什么标准算完成」「什么时候可以上线」。

这三个判断，看起来简单，但在并行项目里极其容易被淹没。

技术同学跑得很快。异乡人才从 2026 年 2 月启动，几周内就跑出了 18,000+ 行代码；异乡点评有 505 个单元测试在跑着守住质量底线；dingning.ai 在 3 月上线后每周都在迭代。每条线都有自己的节奏，而我需要同时在三条线上做出「有质量的决策」。

**工具不是问题。流程才是。**

## 企业微信的「非标用法」

我们公司确认使用企业微信。但我一开始只是把它当 IM 用——发消息、拉群、偶尔审批一个请假。

后来我意识到，企业微信的审批流、群机器人、消息提醒，其实可以被改造成一套「轻量级项目看板」。

我没有把它搭成复杂系统。我只做了三件事：

**1. 审批流 → 决策留痕**

每次重要决策——要不要上这个功能、要不要调整优先级——我都走企业微信审批。不是因为流程主义，而是因为**审批强迫我把判断写下来**。

"这个功能上，理由是什么？标准是什么？谁负责跟进？"

写下来的那一刻，模糊的想法就变成了可执行的指令。而且审批记录留着，两周后谁都不会失忆。

**2. 群机器人 → 进度可见**

技术团队每次有关键动作（PR 合并、功能部署、测试通过），通过群机器人推一条消息到项目群。

我不用盯着每一行代码。我只需要看消息，就能知道「异乡点评今天发生了什么」「异乡人才的推荐引擎跑出结果了吗」。

**3. 消息提醒 → 我的外部大脑**

我在企业微信里设置了定时提醒，充当我自己的「任务提醒器」。每周一早上，我会看到本周三条线各自要推进什么，而不是靠记忆。

这不是高科技。但它让我从「被动接收信息」变成「主动驾驭节奏」。

## AI 介入之后，发生了什么变化

真正让这套系统跑顺的，是 Claude 的加入。

我的用法很具体：**每次要启动一个新功能或新模块，我先跟 Claude 把任务拆解清楚，再把拆解结果推进企业微信审批流。**

拆解的过程是这样的：

- 我告诉 Claude：「异乡人才需要优化顾问评价系统的展示逻辑，目标是让用户更快找到高质量顾问」
- Claude 帮我拆出：前端展示调整、Thompson Sampling 参数验证、AB 测试方案、上线后的数据观测指标
- 我过一遍，删掉不必要的、补充我知道但 AI 不知道的业务上下文
- 最终版本进审批流，变成「可执行的任务单」

这个过程有一个隐藏价值：**它逼我在启动之前就想清楚「完成的标准是什么」**。

没有 AI 辅助拆解之前，我经常在功能做完之后才发现「这不是我想要的」。现在少多了。

## 一次功能上线的全流程

用一个真实的流程结构来说明，异乡点评的一次功能迭代大概是怎么跑的（以实际流程为准，不虚构细节）：

| 阶段 | 工具 | 我做什么 |
|------|------|----------|
| 需求澄清 | Claude | 和 AI 对话，把模糊想法拆解成可执行任务 |
| Gate-Review 评分 | 自定义打分表 | 对功能必要性、复杂度、优先级打分，决定是否启动 |
| 正式立项 | 企业微信审批 | 提审，写清楚目标、验收标准、负责人 |
| 开发中 | 企业微信群 | 群机器人推进度，我异步跟进 |
| PR 合并 | GitHub + 企业微信通知 | 技术同学合并后，群里收到推送 |
| 上线确认 | 企业微信审批 | 走上线审批，我点「同意」= 正式生效 |
| 复盘 | 文档 + 下次 Gate-Review | 数据表现记录，影响下一次优先级判断 |

**Gate-Review** 是我在异乡人才开发过程中摸索出来的方法——在启动一个功能之前，先打一张「值不值得做」的分。评分维度包括：对核心用户的价值、实现复杂度、当前优先级匹配度。

低分功能直接进冷冻区，不让它占用团队资源。

这套门禁机制，配合企业微信的审批留痕，让我的决策变得可追溯、可反驳、可学习。

## 非技术主理人的工具哲学

用了一段时间之后，我总结了一个判断工具好不好的标准：

**它有没有让我做出更好的决策，还是只是让我「管理了更多工具」？**

我见过不少团队，用了很多项目管理软件，但每天花大量时间在「更新看板」本身。工具变成了目的，而不是服务于目的的手段。

我的选择刚好相反：用最少的工具，把「决策节奏」跑顺。

- 企业微信：已有，不额外引入新工具
- Claude：已有，用来思考，不用来执行
- GitHub：技术同学在用，我借用它的 PR 机制做上线门禁

三个工具，覆盖了「想清楚 → 立项 → 跟进 → 上线」的完整链路。

我不需要知道 FastAPI 怎么写路由，不需要懂 NestJS 的 Module 是怎么组织的。但我需要知道：**这件事现在在哪里、卡在什么、下一步谁来推**。

这就是非技术主理人的核心职责。工具是为这个职责服务的，不是反过来。

## 如果你也在跑多个项目

以下几点，不是建议，是我踩过坑之后觉得真正有用的事：

**1. 决策必须留痕**
口头说的永远会失忆。哪怕只是一条企业微信审批、一个文档片段，写下来就是留痕。

**2. 不要靠记忆管任务**
人的工作记忆容量是有限的。外包给系统（哪怕只是定时提醒），让大脑留给判断，不要留给"记住它"。

**3. 在启动前就定义「完成标准」**
这是 Gate-Review 最核心的价值。没有标准，就没有验收，就没有真正的完成。

**4. AI 最好的用途是帮你「想清楚」**
不是帮你执行，是帮你在开始执行之前，把问题拆解清楚。这一步省下来的时间，比任何执行加速都更值钱。
