博客的流量是偶发的。

某篇文章被转发，突然来了几百个访客，然后……消失了。他们不知道你是谁，你也不知道他们是谁。下次你写出更好的东西，他们不会收到任何通知。

这是我在 dingning.ai 上线后很快意识到的问题。**网站是一个容器，但如果没有留住读者的机制，它只是一个流量的过路站。**

Newsletter 是我找到的答案。

---

## 为什么订阅用户才是真正的资产

我做的事情是国际教育 + AI 布道。这个领域的读者有一个特点：他们不是随机刷到你的人，而是真正在思考「AI 怎么改变教育」「如何在海外重建职业身份」这些问题的人。

这样的读者，值得被好好对待。

社交媒体的算法随时可以把你的触达率砍掉一半。博客的 SEO 是长期游戏，短期内很难依赖。**唯一真正属于你的渠道，是一份可以直接发到读者收件箱的 Newsletter。**

订阅列表不属于平台，属于你。这是我决定认真做 Newsletter 的核心逻辑。

---

## 为什么选 Buttondown，而不是 Substack 或 ConvertKit

工具选型是我花时间最多的一步。对比下来，主要考虑了三个维度：

### 定价逻辑

| 工具 | 免费额度 | 收费方式 |
|------|----------|----------|
| Substack | 无限免费 | 抽取付费订阅收入 10% |
| ConvertKit | 1000 订阅免费 | 按订阅人数阶梯收费 |
| Buttondown | 100 订阅免费 | 按订阅人数，起步极低 |

Substack 的商业模式决定了它更适合直接做付费订阅的创作者。我现在还没到那一步，用它有点「大材小用」，而且平台调性更偏向英文媒体。

ConvertKit 功能强大，但对我这个阶段来说，90% 的功能用不上，界面反而让我迷失在设置里。

Buttondown 的产品哲学是：**做一个简单、干净、以写作为中心的工具**。这和我对 Newsletter 的定位高度吻合。

### 数据自主权

Buttondown 支持完整的数据导出，订阅者列表随时可以带走。这一点对我来说不是可选项，而是必须项。

另外，Buttondown 的 API 文档写得非常清楚，这在后面的集成环节省了我很多时间。

### 与 Next.js 的集成简洁度

dingning.ai 用的是 Next.js 14。我需要的是：在网站上放一个订阅表单，用户填邮箱，直接加入我的 Buttondown 订阅列表。

这个流程用 Buttondown 的 API 实现起来非常干净，不需要任何第三方 SDK，一个 `fetch` 请求搞定。

## 用 Vibe Coding 把订阅表单嵌进网站

我不是程序员。dingning.ai 的整个网站，都是用 AI 辅助完成的——这是「Vibe Coding」的核心：你负责提需求和判断，AI 负责写代码。

### 整体思路

Buttondown 提供 REST API。订阅一个新用户，只需要向 `/v1/subscribers` 发一个 POST 请求，携带邮箱和 API Key 即可。

我的实现分三层：

1. **前端组件**：一个简单的邮件输入框 + 订阅按钮，带加载状态和成功/失败提示
2. **Next.js API Route**：作为中间层，避免把 API Key 暴露在前端
3. **Buttondown API**：实际执行订阅操作

### 和 AI 协作的过程

我告诉 Cursor：「帮我在 Next.js 14 里创建一个 Newsletter 订阅组件，调用 Buttondown API，API Key 存在环境变量里，要有 loading 状态和错误处理。」

第一版代码大概 10 分钟出来了。但我没有直接用，而是逐段读懂了逻辑，然后提了几个修改：

- 成功后邮件输入框要清空
- 错误信息要对用户友好，不要暴露技术细节
- 按钮样式要和网站整体的海军蓝保持一致

**这是 Vibe Coding 里我认为最关键的一步：不是让 AI 直接给你答案，而是你参与每一个决策。**

最终这个组件被放在了首页、博客文章底部、以及一个独立的「订阅」页面。三处位置，一个组件，复用。

### 一个小坑

Buttondown 对重复订阅的处理方式：如果邮箱已经在列表里，API 会返回 `400` 错误，但这不代表「出错了」，而是「已经订阅了」。

我最开始直接把 400 显示为错误，后来专门处理了这个状态，提示「你已经订阅了，感谢你的支持」。这个细节处理起来只需要几行代码，但用户体验差很多。

## 冷启动阶段的真实数据

dingning.ai 在 2026 年 3 月 7 日上线，Newsletter 功能随网站一起上线。

### 第一个订阅者

第一个订阅者是我自己。

这不是玩笑。我用自己的邮箱测试了整个订阅流程，确认邮件正常到达、格式没有问题、取消订阅链接有效。这是上线前的最后一个质检步骤。

**第一个真正意义上的「外部」订阅者，来自一个朋友圈的转发。** 我发了一条关于网站上线的朋友圈，有人直接点进来，订阅了。

### 前期增长的来源

我不会用虚构的数字来装点这篇文章。网站刚上线不久，我正在经历每一个 Newsletter 创作者都会经历的冷启动阶段。

目前的增长来源主要是：

- **直接分享**：在朋友圈、微信群发文章链接
- **博客文章末尾的订阅引导**：每篇文章结尾都有一个「如果你觉得有用，订阅 Newsletter 第一时间收到新文章」的提示
- **口耳相传**：有人看了文章，推荐给了自己的朋友

冷启动就是这样的。没有捷径，只有一个一个地积累。

**我更在意的是订阅者的质量，而不是数量。** 能在早期阶段找到真正关心「国际教育 + AI」这个交叉点的读者，本身就是一件很有价值的事。

## Newsletter 和博客的内容协同策略

这是我目前正在摸索的部分，谈谈现阶段的思考。

### 哪些内容先发邮件

- **过程性记录**：比如「我今天用 AI 做了一个新实验，结果出乎意料」——这类内容时效性强，适合先通过邮件发给订阅者，给他们一种「内部消息」的感觉
- **不成熟的想法**：博客文章需要打磨，但邮件可以更随意一些，像写信一样

### 哪些内容直接上博客

- **经过验证的方法论**：比如这篇关于 Buttondown 的文章，是在完成整个搭建过程之后才写的，有完整的来龙去脉
- **工具测评和教程**：这类内容有较长的参考价值，适合留在博客里被搜索引擎收录
- **项目复盘**：适合公开发布，也方便分享传播

### 内容协同的核心原则

**不要让博客和 Newsletter 互相复制。**

最理想的状态是：订阅者在邮件里看到的是你在「思考过程中」，博客读者看到的是你「思考完成后」的输出。两者互为补充，而不是重复。
