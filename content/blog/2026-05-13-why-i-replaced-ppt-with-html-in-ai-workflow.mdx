三月底，我在济南香格里拉参加了一场 AI 主题的开放麦活动。

轮到我上台，我打开的不是 PowerPoint，不是 Keynote，而是一个本地 HTML 文件。浏览器全屏，键盘左右键翻页，效果和 PPT 没什么两样——但背后的逻辑，完全不同。

有人会后问我：「你用的什么工具做的？」

我说：「Claude Code，十分钟。」

对方停顿了一下，然后笑着说：「我以为你会说 Gamma。」

---

## PPT 在 AI 工作流里「断路」了

我不是要黑 PPT。PowerPoint 和 Keynote 都是成熟的工具，在它们擅长的地方——精细排版、企业模板、多人协作审阅——依然无可替代。

但当你开始认真使用 AI 来辅助内容创作，你会发现 PPT 和 AI 之间有一道隐形的墙。

**第一，AI 读不了 PPT 的内容结构。**

你想让 Claude 帮你改一张幻灯片的文案，你得先截图、或者手动复制文字，再粘贴过去。它看到的是「一段文字」，而不是「第三张幻灯片的标题和三个要点」。结构在 .pptx 的二进制格式里，AI 摸不到。

**第二，PPT 不能被版本管理。**

Git 对二进制文件基本没辙。你改了一版演示，只能另存为「终版」「终终版」「真的终版」。如果你像我一样，把所有内容都放在 Git 仓库里管理，PPT 是一个永远格格不入的异类。

**第三，PPT 是静态的，嵌不进实时内容。**

我在济南那场演讲的主题是 AI 工具实践。我想在演示里放一个可以现场交互的 prompt 示例，想嵌一段代码高亮，想让某个数据展示带点动效。PPT 能做，但那个复杂程度，远超我愿意付出的时间。

HTML 能做，而且 AI 可以帮我写。

---

## 用 Claude Code 生成演示文件的真实过程

我在会议前一天晚上才开始准备演示材料。这是我的真实步骤。

### 第一步：用 Markdown 打草稿（约 15 分钟）

我先在编辑器里写了一份 Markdown 大纲，大概是这样的结构：

```
# AI 工具实践分享

## 我们在做什么
- 异乡好居：海外华人租房服务
- 合伙人视角的 AI 实验

## 核心论点
- AI 不是替代工具，是协作者

## 三个真实案例
...
```

Markdown 写起来没有心理负担，不用考虑字体、配色、排版。想到什么写什么，结构对了就行。

### 第二步：丢给 Claude Code，提一个清晰的需求（约 3 分钟）

我的 prompt 大概是：

> 把这份 Markdown 大纲转成一个单文件 HTML 演示，要求：深色背景，代码块有语法高亮，左右键翻页，每张幻灯片内容不要太满，标题大一点，底部显示页码。不需要任何外部依赖，所有 CSS 和 JS 内联。

关键在最后那句——**所有资源内联到单个 HTML 文件**。这样我不需要网络，不需要服务器，任何浏览器打开就能用。

### 第三步：看效果，微调（约 5 分钟）

Claude Code 跑完以后，我直接在浏览器里打开 HTML 文件预览。发现某张幻灯片字太多，告诉它「第三张幻灯片拆成两张」，它改完，我再刷新。

整个过程，从提需求到满意的版本，确实没超过十分钟。

第二天我就带着这个文件去了济南。全程没有 PPT 相关的软件依赖，也没有「字体缺失」「版本不兼容」这类烦人的意外。

## HTML 演示的局限：不是银弹

说完好处，我必须说清楚它的边界，否则这篇文章就是在骗人。

**不适合 HTML 演示的场景：**

- **多人协作修改**：HTML 文件不是为协同编辑设计的，如果需要三个同事同时改一份演示，还是用 Google Slides 更合适
- **复杂图表和数据可视化**：虽然 HTML 理论上能做任何图表，但如果你要对接公司 BI 系统的数据，或者做精细的图表调整，专业工具的成本更低
- **企业正式汇报**：有些甲方或者领导就是习惯看 PPT 格式，这是职场现实，硬用 HTML 反而增加不必要的摩擦
- **需要打印或导出 PDF**：HTML 的打印排版需要额外处理，不如 PPT 直接导出来得方便

**适合 HTML 演示的场景：**

- 技术分享、内部培训、社区开放麦
- 需要嵌入代码、动效、或者实时内容的场合
- 你想把演示文件放进 Git 仓库版本管理
- 你要快速生成一份「够用但不需要精美」的演示

我在济南的那场开放麦，完美落在适合的场景里。如果是给客户做正式商务提案，我可能还是会打开 Keynote。

## 我现在的演示三件套

经过这几个月的实践，我的演示准备流程已经固定下来：

**① Markdown 打草稿**

把大脑里的内容先倒出来，不管格式。用任何顺手的编辑器，几百字的提纲，十分钟以内。

**② AI 生成 HTML**

把 Markdown 和需求一起丢给 Claude Code，告诉它演示的风格、受众、特殊要求。生成完看一遍，有问题直接对话调整。

**③ Vercel 预览链接分享**

如果需要提前分享给组织方或者观众，我会把 HTML 文件推到 GitHub，Vercel 自动部署，生成一个预览链接。对方不需要下载任何文件，手机打开链接就能看。

这套流程最大的优点是：**每一个环节都是纯文本，AI 全程可以介入**。从草稿到成品，我不需要切换到任何「AI 不懂的」专有格式。

---

## 为什么这件事值得认真对待

我写这篇文章不是为了推销 HTML。

我想说的是：**当你把 AI 真正引入工作流，不只是偶尔问一下问题，而是让它参与内容生产的全链路——你会开始重新审视每一个工具的选择标准。**

不再是「这个工具功能强不强」，而是「AI 能不能在这个工具的上下游顺畅工作」。

PPT 功能很强。但它在 AI 工作流里制造了断点。

HTML 没有那么强。但它在 AI 工作流里是透明的、可读的、可生成的。

济南那场开放麦结束以后，有几个人加了我微信，说想了解「怎么真正把 AI 用进日常工作」。这个问题没有一个通用答案，但我的经验是：从最具体的场景开始改变——比如，下一次做演示的时候，试试不打开 PPT。

---

*工具的选择，本质上是思维方式的选择；你用什么做演示，折射出你怎么看待 AI 在你工作里的位置。*
