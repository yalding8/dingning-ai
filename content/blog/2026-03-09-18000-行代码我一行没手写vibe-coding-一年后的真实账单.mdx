一年前，我做了一个在朋友圈很难解释的决定。

39岁，没有任何编程背景，我说要用 AI 写一个真实上线、真实运营的产品——不是 demo，不是玩具，是能给真实用户用的东西。

身边没人觉得这个想法靠谱。

一年后，我坐在这里，面前是两个落地产品、18000+ 行代码，和一份说不清是骄傲还是后怕的复盘心情。

---

## 一年前的赌注：39岁，决定认真的

我不是第一次"尝试编程"。Python 教程看过，Coursera 课上过，每次都死在第三周。

但这次不一样。

2024 年初，我开始做**异乡人才**——一个帮留学生找回国工作机会的 AI 求职平台。这不是兴趣项目，是我判断真实存在市场需求的产品。我没有钱雇开发，也不想等。

与此同时，我还在搭建自己的个人网站 dingning.ai，作为国际教育 AI 布道者的品牌阵地。

两个项目，两种挑战，一个工具：AI。

我给自己定了一个规则：**我不学编程语法，但我要学会"指挥"AI 写代码。** 区别很微妙，但很关键。

---

## 真实账单：18000 行代码背后的成本

先上数据，再讲故事。

| 维度 | 异乡人才 | dingning.ai |
|------|----------|-------------|
| 代码行数 | ~15,000 行 | ~3,000 行 |
| 上线时间 | 约 5 个月 | 约 6 周 |
| 活跃岗位数 | 10,645 个 | — |
| 单元测试数 | 505 个 | — |
| 手写代码 | 0 行 | 0 行 |

**时间成本**：比我预期多 2 倍。不是因为 AI 慢，而是因为**我**慢——慢在理解业务逻辑，慢在描述清楚我想要什么，慢在出了 bug 不知道从哪里开始排查。

**工具成本**：Claude、Cursor、GitHub Copilot，每月加起来大概 200-300 元人民币。这笔钱花得值，但不是零成本。

**心理成本**：这是账单里最贵的一项，也是没人告诉你的部分。

有那么几次，我盯着一堆报错信息，完全不知道发生了什么。AI 给了我五个"可能的解决方案"，我一个个试，第六次才对。那种感觉不是挫败感，是一种很奇特的**失控的控制感**——你在推进，但你不知道自己在往哪里推。

## 三个关键时刻

### 推荐引擎：AI 救了我一次

异乡人才的核心功能是岗位推荐。最早版本很粗暴：按发布时间排序，加几个标签过滤。

用户体验很差。

我问 Claude：有没有比"平均分排序"更公平的推荐算法？它给我介绍了 **Thompson Sampling**——一种用于解决"探索-利用困境"的概率算法，常用于推荐系统和 A/B 测试。

我没学过统计学。但 AI 用"你如何在餐厅菜单上选新菜"这个类比把逻辑讲清楚了。我听懂了，然后让它写代码，让它解释每一行在做什么，直到我真的理解了整个机制。

这是 Vibe Coding 最神奇的地方：**你不需要先学会，你可以在做的过程中理解。**

### 505 个单元测试：差点坑了我的那次

推荐引擎上线一周后，有用户反馈结果"感觉不对"。

我打开代码，完全不知道从哪里开始 debug。

那次教训很深刻。后来我让 AI 帮我为 Thompson Sampling 的排名算法补写单元测试——最终写了 505 个。

505 个不是炫耀，是**恐惧的量化**。

每次我对推荐逻辑做任何修改，测试套件会告诉我有没有破坏什么。这让我敢于迭代，否则我只会原地不动，因为改了怕出问题。

**单元测试是 Vibe Coding 的安全网。没有它，你只是在走钢丝。**

### Gate-Review：我自己发明的质量门禁

第三个关键时刻不是一次危机，而是一个系统。

做到一半，我发现自己陷入了一种很危险的节奏：AI 给什么就用什么，功能叠功能，没有人检查整体方向有没有跑偏。

我引入了 **Gate-Review 评审方法论**：每完成一个功能模块，在进入下一阶段前，强制做一次"产品评审"——不是代码审查，是业务逻辑审查。

- 这个功能解决了真实问题吗？
- 它和整体产品方向一致吗？
- 它引入了我没能力维护的复杂度吗？

很多时候，这个环节让我删掉已经写好的功能。这不是浪费，这是**最便宜的质量控制**。

## Vibe Coding 的天花板在哪里

说了这么多好处，说说边界。

**我能做到的：**
- 完整的 Django 后端 + React 前端
- PostgreSQL 数据库设计和查询优化
- 自动化爬虫和数据更新流程
- 复杂算法的理解和应用（在 AI 的帮助下）
- 生产环境部署和监控

**我做不到的：**
- 真正理解为什么某些性能问题发生
- 在没有 AI 的情况下独立 debug 复杂的系统级错误
- 评估架构决策的长期代价（这需要经验，不是工具）
- 快速适应 AI 完全陌生的技术栈

有一次，我遇到一个 Redis 缓存的竞态条件（race condition）问题。AI 解释了原因，给了方案，但我折腾了两天才真正理解发生了什么。

**Vibe Coding 不是魔法，它是一个杠杆。杠杆放大你的能力，但你得先有能力。**

对非程序员来说，这个"能力"不是代码能力，而是：
- 清晰描述问题的能力
- 分解复杂需求的能力
- 识别 AI 在胡说的能力（这个最难学）

## 给下一个想用 AI 写产品的非程序员，三条真心话

### 第一条：从真实需求出发，不要从技术出发

我见过很多人的 Vibe Coding 项目死在"我想用 Next.js 做一个……"这句话上。

正确的起点是："**有一个真实的问题，我需要一个产品来解决它。**"

技术选型交给 AI，你只需要描述你的用户是谁，他们的痛点是什么。

### 第二条：学工程实践，不学编程语法

Git、单元测试、代码评审、部署流水线——这些东西和编程语言无关，但它们决定了你的产品能不能活过第一个月。

我一开始没有认真做 Git 版本管理。有一次我的一个"优化"把数据库查询性能拖垮了，我不知道是哪一步改的，花了三天才找到问题。

**版本控制是时光机。没有它，你在裸奔。**

### 第三条：做好"慢下来"的心理准备

Vibe Coding 最大的谎言不是"AI 会出错"——所有人都知道这一点。

最大的谎言是"Vibe Coding 很快"。

它在某些阶段确实很快，比如搭骨架、写重复性代码、处理样式细节。但每当你进入真正有业务复杂度的领域，**速度会掉下来，而且你很难预测它会慢多少。**

做异乡人才的推荐引擎，我估计一周，实际花了三周。

这不是失败，这是学习的代价。把它算进你的时间预算，而不是在焦虑中把它当成信号。

## 一年后，我是什么感觉

两个产品上线运营，10000+ 真实岗位，505 个测试保障的算法，一年积累的七篇博客文章。

我没有变成程序员。我也不打算变成程序员。

但我变成了一个**能够独立思考、独立构建、独立负责**一个技术产品的人。这件事在一年前，我真的没有想到是可能的。

Vibe Coding 改变的不只是我写产品的方式，它改变的是我对"我能做什么"这个问题的边界感知。

如果你也在 39 岁、或者任何年龄，站在那个"要不要试试"的路口——

我不会告诉你它很容易。但我会告诉你：**它比你现在想的，要更值得。**
