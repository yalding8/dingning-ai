做技术选型这件事，听起来很理性——对比参数、跑 benchmark、算成本。但实际上，很多时候最终的决定来自于一个很朴素的原因：用着顺手，出了问题有人兜底。

异乡点评是我在异乡好居体系下推进的一个顾问评价平台，面向的用户主要在国内。这个项目的生产服务器，从一开始就跑在[阿里云](https://www.aliyun.com)上。今天聊聊为什么，以及用下来的真实感受。

## 国内业务的第一选择

这个判断没什么悬念。如果你的用户在中国大陆，服务器也必须在中国大陆。备案、CDN 加速、网络质量，这些事情国内的云厂商天然有优势。而在国内的几家主流云服务里，阿里云的市场份额和技术成熟度确实排在前面。

异乡点评的架构是 Node.js 后端 + PostgreSQL + Redis，用 Docker 容器化部署在 ECS 上。整套环境搭起来并不复杂，但稳定性要求不低——平台上有真实的顾问评分数据，服务中断会直接影响用户信任。

跑了这么长时间，ECS 实例的可用性没让我失望过。偶尔做系统维护时阿里云会提前通知，给足迁移时间，这种确定性对于小团队来说很重要。

## Docker 部署的体验

异乡点评的部署流程是标准的容器化方案：本地构建镜像，推到服务器上用 docker-compose 编排启动。阿里云的 ECS 对 Docker 的支持很成熟，内核版本、存储驱动这些底层的东西不需要额外折腾。

我在 docker-compose 里配了 healthcheck 和自动重启策略，加上 pg_dump 定时备份脚本，整个运维负担其实很轻。大部分时间我根本不需要登录服务器，除非是做版本更新或者排查偶发问题。

阿里云容器镜像服务（ACR）我也用过一段时间，私有镜像仓库对于不想把代码推到 Docker Hub 的项目来说很实用。不过异乡点评的规模还不需要 Kubernetes，一台 ECS 加 Docker Compose 就够了。

## 不只是一台服务器

说到[阿里云](https://www.aliyun.com)的生态，这可能是它和海外云服务拉开差距的地方——至少在国内场景下是这样。

**对象存储 OSS**：异乡点评的静态资源、用户上传的图片都存在 OSS 上。SDK 好用，价格低廉，配合 CDN 加速后国内访问速度很快。

**云数据库 RDS**：虽然异乡点评目前用的是 ECS 上自建的 PostgreSQL，但我在其他项目里用过 RDS。自动备份、主从切换、慢查询分析这些功能开箱即用，适合不想花精力管数据库的团队。

**日志服务 SLS**：做异乡点评的评分算法调优时，我需要分析大量的请求日志和评分计算日志。SLS 的查询语法虽然有学习成本，但一旦熟悉了，排查问题的效率确实高。

**安全组和防火墙**：阿里云的安全组配置比较直观，端口管理、IP 白名单这些基础安全策略几分钟就能搞定。

## 控制台的复杂度

公平地说，阿里云的控制台不算简洁。产品线太多，菜单层级太深，新用户可能会有些迷失。和 DigitalOcean 那种极简风格比起来，阿里云的后台更像是一个企业级工具——功能强大，但上手成本也更高。

另外，阿里云的计费模式比较复杂，包年包月、按量付费、抢占式实例，各种优惠活动叠加在一起，有时候算清楚到底要花多少钱本身就是一个挑战。我的建议是：小项目直接选轻量应用服务器或者包年的 ECS，不要在优化成本上花太多时间。

## 什么场景下我会推荐阿里云

结合我自己的使用经验，以下场景阿里云是比较好的选择：

- 用户主要在中国大陆，需要备案和国内 CDN
- 项目使用容器化部署，需要成熟稳定的 Linux 环境
- 业务会用到对象存储、数据库、消息队列等多种云服务，希望在一个平台上搞定
- 团队需要企业级的安全合规能力

异乡点评从开发到上线，阿里云一直是稳定的底座。它不是最便宜的，也不是最酷的，但对于国内业务来说，它是那个你不太需要担心的选择。

如果你也在做面向国内用户的项目，不妨去[阿里云官网](https://www.aliyun.com)了解一下各类云服务的定价方案，根据自己的业务规模选择合适的配置。
