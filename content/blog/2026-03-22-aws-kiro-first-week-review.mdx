我是那种会在凌晨两点对着屏幕问 AI「你能帮我写一个 FastAPI 路由吗」的人。

不是程序员。是异乡好居的合伙人，同时是 dingning.ai 的主理人。用 Vibe Coding 的方式，在过去一年多里陆陆续续把几个产品从零做起来。

市面上评测 AWS Kiro 的文章，大多在聊 agentic loop、spec-driven development、benchmark 对比。我看完基本看不懂，但我有一个很朴素的问题：

**它有没有让我少求助 AI 三次？**

用了一周之后，我有答案了。

---

## AWS Kiro 是什么，它和 Claude Code 有什么不同

先简单说清楚背景，方便没接触过的朋友理解。

AWS Kiro 是亚马逊推出的 AI 编程工具，目前处于 preview 阶段。它不是一个对话框，更像一个「会主动思考的开发搭档」——你给它一个需求，它会先帮你梳理需求文档（spec），再拆解任务，然后逐步实现。

我之前主要用 Claude Code。Claude Code 更像一个随叫随到的高手朋友：你描述问题，它直接上手写，效率极高，但**它不会主动问你「你想清楚了吗」**。

Kiro 和它最大的区别，在我看来是这两点：

| 维度 | Claude Code | AWS Kiro |
|------|-------------|----------|
| 工作方式 | 对话驱动，你说它做 | 先写 spec，再执行 |
| 适合场景 | 明确任务、快速出活 | 需求模糊、功能复杂 |
| 对非程序员友好度 | 高，但需要你会提问 | 更高，它会帮你想清楚 |
| 与 AWS 生态整合 | 无 | 深度整合 |

我们公司本来就在用 AWS，所以 Kiro 对我来说没有「要不要接入」的问题，只有「好不好用」的问题。

---

## 我拿哪个项目练手，解决了什么问题

今年 2 月，我启动了异乡人才项目——一个帮留学生找回国就业机会的平台，目前有 10,645 个活跃岗位。后端是 Python + FastAPI，前端是 React，数据库 PostgreSQL + Redis，另外还有一个用 Thompson Sampling 做的智能推荐引擎。

整个项目是我一个人用 Vibe Coding 做起来的，代码量达到了 18,000+ 行。

听起来很厉害，实际上踩了很多坑。

最让我头疼的不是写代码，而是**需求没想清楚就开始做**。比如推荐引擎那块，我一开始只说「帮我做一个推荐算法」，Claude Code 确实很快给了一个实现，但等我跑起来之后才发现，它的排序逻辑和我脑子里想的完全不一样——我想要的是「越少数据也能给出合理推荐」，而不是简单的评分平均。

这种情况，来回修改了好几轮，浪费了大量时间。

Kiro 介入的时机，就是在这类「我知道我要什么，但说不清楚」的场景。

---

## 上手一周的真实体验：惊喜和踩坑

### 惊喜一：它会逼你想清楚

用 Kiro 处理一个新功能的第一步，它会生成一份 spec 草稿——把你的需求翻译成结构化的文档：目标是什么、边界在哪里、可能的异常情况有哪些。

第一次看到这个，我愣了一下。

因为它列出来的边界情况，有两三条是我**完全没想到**的。比如「当候选人的学历字段为空时，推荐引擎应该如何处理？」这种问题，我之前靠直觉跳过去了，结果上线后真的踩坑了。

**Kiro 的 spec 流程，本质上是在替你做产品评审。** 对非程序员来说，这个价值被严重低估了。

### 惊喜二：任务拆解让我不再「盯着一整座山发呆」

非程序员做复杂功能，最容易卡住的不是技术，是**不知道从哪里下手**。

Kiro 在你确认 spec 之后，会自动把任务拆成一个 checklist：先做哪个模块，依赖关系是什么，哪些可以并行。

这对我来说相当于有人帮我做了任务管理。我不需要再在脑子里维护一张混乱的 to-do，只需要跟着走就好。

### 踩坑一：spec 生成有时候过于「正式」

有几次，Kiro 生成的 spec 写得像企业级 PRD，字段定义、数据流图、接口规范……我一个人做的小功能，根本用不上这个粒度。

需要手动告诉它：「简洁一点，我不需要完整文档，只需要思路确认。」它会调整，但**默认值偏重**，不适合快速迭代的小改动。

### 踩坑二：和现有代码的上下文理解，还有差距

异乡人才这个项目代码量不小，Kiro 在处理跨文件的逻辑时，偶尔会「失忆」——它理解了 A 文件的逻辑，但没有意识到 B 文件里有一个相关的函数已经写过了，导致重复实现。

这个问题 Claude Code 也有，但感觉 Kiro 因为 spec 流程本身就更重，一旦上下文丢失，纠正起来要多花一些时间。

### 踩坑三：启动成本真的高一点

如果你只是想「快速加一个字段」或者「改一行逻辑」，Kiro 的 spec-first 流程会让你觉得很繁琐。

**小改动还是 Claude Code 更顺手。Kiro 更适合「我要做一个新功能」这个量级的任务。**

## 它在我开发流程里处于什么位置

用了一周之后，我的工具分工大概长这样：

| 场景 | 用什么 |
|------|--------|
| 快速修 bug、改逻辑 | Claude Code |
| 新功能从零开始 | Kiro（写 spec）→ Claude Code（执行） |
| 代码审查、解释逻辑 | Claude Code |
| 需求还没想清楚时 | Kiro（让它帮我想清楚） |

对，你没看错——我的用法是**Kiro 负责想，Claude Code 负责写**。

Kiro 的 spec 输出本身就是很好的提示词素材。我有几次直接把 Kiro 生成的需求文档贴给 Claude Code，后者的实现质量明显比我自己描述时高出一个档次。

两个工具不是竞争关系，而是互补的。

---

## 非程序员视角的总结：值不值得切换

先说结论：**不是切换，是叠加。**

如果你已经在用 Claude Code 或者其他 AI 编程工具，Kiro 不会替代它们。但如果你和我一样，是非程序员、靠 Vibe Coding 做产品，Kiro 填补了一个我之前一直缺失的环节：

**在开始动手之前，先把事情想清楚。**

这听起来很朴素，但我在异乡人才开发过程中因为「没想清楚就开始做」浪费的时间，少说也有两三周。Kiro 的 spec 流程，某种程度上在替我做这件事。

一周下来，我给它的评分是：

- **上手难度**：★★★☆☆（比 Claude Code 略难，主要是要适应 spec 流程）
- **对复杂需求的帮助**：★★★★★
- **对快速小改动的帮助**：★★☆☆☆
- **与 AWS 生态整合**：★★★★☆
- **总体值不值**：★★★★☆

如果你在做一个有一定规模的项目，有时候会在「从哪里下手」这件事上卡壳，我建议你试试。反正还在 preview 阶段，现在入手的成本不高。

如果你只是偶尔写写脚本，不用特地换工具。

*工具是手段，但想清楚问题本身，才是非程序员做产品最难跨过的那道坎。*
