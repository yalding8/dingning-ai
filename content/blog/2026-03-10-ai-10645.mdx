## 1. 从痛点出发：为什么一个国际教育 VP 要自己动手写求职平台？

大家好，我是 Ning Ding。我的主业是国际教育，一个典型的“非程序员”。每天打交道的是学生、家长、学校，代码离我的世界似乎很远。

直到我看到了一个让我无法忽视的“痛点”。

在我的工作中，我接触了太多优秀的留学生。他们带着全球视野和扎实的专业知识回国，却在求职时陷入迷茫：**信息碎片化、匹配效率低、缺乏针对性的指导**。市面上的大型招聘平台，对他们而言如同大海捞针；而一些高端求职服务，价格又令人望而却步。

一个念头越来越强烈：能不能为他们做一个专属的、智能的、高性价比的求职平台？

找外包？沟通成本高，需求理解容易偏差，后续迭代更是麻烦。招技术团队？对早期项目来说，成本和时间都是巨大挑战。

就在那时，AI 编程工具（我后来称之为 **Vibe Coding** 的实践方式）进入了我的视野。我决定，自己动手。

**驱动我的不是对技术的热爱，而是对解决一个真实问题的执着。** 我想证明，在 AI 的辅助下，一个具备产品思维和逻辑能力的业务人员，完全有可能将想法直接转化为可运行的产品。

“异乡人才”（UHomes Talent）这个项目，就这样从一个 Excel 表格里的需求清单，开始了。

## 2. 技术栈选择：为什么是 Django + React + PostgreSQL 这个组合？

作为一个技术门外汉，技术选型的第一步不是比较性能，而是 **“学习友好度”和“生态成熟度”**。

我问了自己几个问题，也问了 AI 助手很多遍：
1.  **我想快速构建一个稳定、安全的后台管理系统（比如职位、用户、申请的管理）** -> 答案指向了 **Django**。它的“开箱即用”特性太吸引人了：自带管理员后台、强大的 ORM、严谨的安全防护。我不需要从零开始造轮子，而是站在巨人的肩膀上。
2.  **我希望前端体验是流畅、现代的** -> 答案指向了 **React**。组件化的思想和我设计产品功能模块的思路不谋而合。丰富的社区资源和组件库，让我能像搭积木一样构建界面。
3.  **我的数据关系明确（用户、职位、申请、评价等），且需要可靠存储** -> 答案指向了 **PostgreSQL**。它稳定、功能强大，对 JSON 数据的支持也让我在未来处理一些灵活字段时游刃有余。

这个组合（Django + React + PostgreSQL）就像一个 **“黄金三角”**：
*   **Django** 作为大后方，稳如磐石，处理所有业务逻辑和数据。
*   **React** 作为前哨站，灵活敏捷，负责所有用户交互。
*   **PostgreSQL** 作为仓库，结构清晰，安全地保管所有数据。

更重要的是，关于这个组合的教程、问答、开源项目在互联网上浩如烟海。这意味着，我遇到的 90% 的基础问题，都能通过搜索和询问 AI 找到答案。**对于非程序员来说，选择主流、成熟的技术栈，就是在为自己铺设一条“少踩坑”的高速公路。**

## 3. 核心算法突破：Thompson Sampling 推荐引擎的平民化实现之路

平台的核心价值是“智能匹配”。最初，我设计了一个简单的评分系统：根据学历、专业、技能等维度加权打分，然后按总分排序推荐。

但很快，我发现了一个致命问题：**“马太效应”**。高分顾问会获得更多曝光和申请，从而获得更多好评，分数越来越高，新人或暂时评分不高的优质顾问则永无出头之日。这既不公平，也降低了平台的匹配效率。

我需要一个能 **平衡“探索”（发现新的、有潜力的顾问）和“利用”（推荐已知的优秀顾问）** 的算法。AI 助手向我介绍了多臂老虎机问题和 **Thompson Sampling** 算法。

听到这个名字，我头都大了。但当我让 AI 用最通俗的语言解释时，我理解了它的精髓：

> **想象一下，你面前有10台老虎机（顾问），每台的赢钱概率（真实水平）未知。Thompson Sampling 不会只盯着目前赢钱最多的那台猛拉，而是会根据当前有限的尝试结果，为每台机器计算一个“赢钱概率的可能分布”。每次选择时，它从这个分布里随机采样一个值，然后选择采样值最高的那台去玩。这样，水平高的机器被选中的概率会越来越大，但水平不确定的机器也始终有机会被探索。**

这完美契合了我的需求！我不需要知道顾问的“真实分数”，我只需要一个能动态平衡探索与利用的机制。

**实现过程，是 Vibe Coding 的典型体现：**
1.  **理解意图**：我向 AI 清晰地描述业务问题——“如何避免评分高的顾问垄断流量，让新顾问也有公平曝光机会？”
2.  **获取概念**：AI 提供 Thompson Sampling 作为解决方案，并用比喻让我理解。
3.  **拆解实现**：我要求 AI 将算法拆解为 Django 模型中可以执行的步骤：
    *   每个顾问有“成功”次数（如收到简历后学生给予好评）和“总尝试”次数（被推荐的次数）。
    *   每次推荐时，为每个符合条件的顾问，从其 Beta 分布（由成功次数和失败次数参数化）中随机采样一个值。
    *   按照采样值从高到低排序，生成推荐列表。
4.  **编写与调试**：在 AI 的辅助下，我写出了不到 50 行的核心推荐函数。然后通过大量模拟数据测试，观察其行为是否符合“探索与利用”的预期。

**这个过程让我深刻认识到：非程序员理解复杂算法的关键，不在于推导公式，而在于理解其解决的业务问题（Why）和核心思想（What）。** 具体的实现（How），可以交给 AI 作为翻译和助手。

## 4. 质量保障：505 个单元测试如何成为非程序员的安全网

写代码，尤其是用 AI 辅助写代码，最让人心虚的就是：**它真的对吗？** 今天改了点东西，会不会把昨天好用的功能搞崩了？

尤其是像 Thompson Sampling 这样的概率算法，肉眼很难验证。为了解决这个焦虑，我为自己织了一张 **“安全网”——单元测试**。

我为推荐引擎编写了 505 个单元测试。这些测试覆盖了：
*   **基础逻辑**：给定固定的成功/尝试次数，采样排序是否基本稳定？
*   **边界情况**：一个新顾问（0次尝试）是否会被包含在推荐中？
*   **概率特性**：运行成千上万次模拟，观察优秀顾问的长期曝光率是否更高，新顾问是否持续获得探索机会。
*   **集成测试**：从用户请求开始，到最终返回推荐列表，整个流程是否畅通。

**这 505 个测试，成了我“胆大心细”的底气：**
*   **重构的勇气**：当我觉得代码结构不好，想重构时，跑一遍测试。只要测试全绿，我就知道核心逻辑没坏。
*   **迭代的信心**：我想优化算法或增加新规则，修改后跑测试。如果有测试变红，我能立刻知道是修改引入了 bug，还是测试本身需要更新。
*   **交付的凭证**：这是我能向潜在用户证明“这个智能推荐系统可靠”的最硬核证据。

对于 Vibe Coding 而言，**单元测试不是可选项，而是必选项。** 它是你作为非程序员，在代码的海洋中航行时，手中最可靠的罗盘和救生衣。我采用的 **Gate-Review 方法论** 很简单：任何代码提交前，必须通过所有相关测试。这形成了一个最小的质量门禁。

## 5. 从 0 到 10645：平台上线后的真实数据与迭代思考

平台上线后，数据开始自己说话。目前，我们已经积累了 **10,645 个活跃岗位**，并且这个数字通过自动化抓取每天都在更新。

**一些真实的观察和迭代：**
*   **Thompson Sampling 的效果是可见的**：我们确实观察到新的、评价数据还不多的顾问，获得了合理的曝光并成功匹配。后台的算法日志显示，探索行为在持续发生。
*   **单元测试抓住了线上 Bug**：有一次，在修改一个与推荐无关的页面时，一个隐蔽的改动意外影响了数据查询链。部署前，测试套件中的几个相关测试立刻失败，避免了问题上线。
*   **非程序员开发的优势**：因为我自己就是深度用户和业务负责人，我能以“第一性原理”直接、快速地调整产品。比如，我们根据学生反馈，迅速增加了“岗位收藏夹”和“申请进度跟踪”功能，从构思到上线，周期极短。

**这个项目带给我的，远不止一个平台：**
1.  **思维转变**：我学会了用“可计算”、“可自动化”的视角看待业务问题。
2.  **能力延伸**：我不再只是一个提需求的人，我成为了一个能亲手将需求“编译”成产品的人。我与技术团队的沟通效率也产生了质的飞跃。
3.  **信心建立**：复杂如推荐算法，系统设计如缓存（我们用了 Redis 来提升推荐列表加载速度），只要拆解得当，都能在 AI 的辅助下攻克。

**最后，我想对每一个有想法但被技术门槛吓住的朋友说：**

今天的 AI 工具，已经极大地压缩了“想法”与“产品”之间的距离。你不需要成为全栈工程师，但你需要成为 **“解决问题的架构师”** 。你的核心价值在于：
*   **精准定义问题**
*   **合理拆解模块**
*   **严谨测试验证**
*   **持续收集反馈**

从零到一万，从一行代码到一万多个岗位，这条路的关键不是技术神迹，而是 **将业务逻辑转化为机器逻辑的耐心、借助工具的学习能力，以及用测试保障每一步的严谨**。

技术不再是高墙，而是一套你可以学习的“超级杠杆”。拿起它，从你最痛的那个点开始，撬动你的第一个项目吧。

*世界属于有想法的人，更属于那些能亲手将想法实现的人。*
