我是那种打字很快的人。

快到什么程度？在搭建「异乡人才」平台的高峰期，一天能跟 Claude 来回几十轮对话，手速从来不是瓶颈。**但有一天我意识到，问题从来不是手速——而是脑子里的想法根本没能完整地「出来」。**

## 打字越快，丢失越多

这件事我观察了挺久。

每次我坐在键盘前，准备描述一个需求，脑子里其实已经有一幅完整的图——推荐算法的边界条件、UI 交互的细节、数据结构的取舍理由……但落到键盘上，我会开始「节省」。

- 太长了，删掉一半
- 这个背景 AI 应该能猜到，跳过
- 先发出去看看效果，不行再补充

结果就是：一个在我脑子里清晰无比的需求，到了 prompt 里变成了模糊的三行字。AI 照着这三行字输出，我看了摇头，再补充，来来回回，把节省的时间全还回去了，还带着利息。

认知疲劳是真实的。长时间高强度打字、切换上下文、反复修改 prompt——这些加在一起，让我到下午四点就开始「划水」。不是懒，是脑子已经在保存电量了。

## 我把「说话」接进了 Vibe Coding

2026 年 3 月初，我在搭建 dingning.ai 个人网站。博客系统、MDX 内容管理、Vercel 部署，这些本身不复杂，但我需要频繁地跟 AI 描述需求、调整方向、生成注释文档。

就在那段时间，我开始用 **Typeless** 做语音输入，接进了我的 Vibe Coding 工作流。

具体是这样用的：

**场景一：需求描述**
不打字，直接对着麦克风说。「我想在博客列表页加一个按分类筛选的功能，分类数据从 MDX frontmatter 里读，默认展示全部，点击某个分类高亮显示，URL 不需要变化，纯前端状态管理就好。」

这一段话我说了大概 15 秒。如果打字，我大概会写成：「帮我加一个分类筛选。」

**场景二：prompt 撰写**
遇到复杂的架构问题，我会先口述一遍我的思路，让 Typeless 转录，再稍作整理就发给 AI。语音逼我说完整，而不是说一半就发出去。

**场景三：注释生成**
AI 帮我写完一段代码，我口述功能说明，转成文字，直接让 AI 填进注释。省掉了「我该怎么描述这段逻辑」的思考成本。

## 键盘 vs 语音：一次真实对比

我用同一个需求做过对比。背景是「异乡人才」的顾问推荐排序逻辑——我需要向 AI 解释为什么用 Thompson Sampling，以及边界条件是什么。

**键盘版（我实际发出去的 prompt）：**

> 帮我优化顾问推荐排序，用 Thompson Sampling，样本少的要往前排。

**语音版（转录后的 prompt，只做了标点整理）：**

> 我们的顾问推荐目前是用平均评分排序，但问题是新顾问评价太少，平均分容易虚高或虚低，结果就是新顾问一直排在后面，没有曝光机会。我想改成 Thompson Sampling，每次展示前根据 Beta 分布采样，置信区间宽的顾问——也就是评价少的——采样结果方差更大，有机会排到前面。边界条件是：如果顾问上线不足 7 天，强制进入随机池，不参与 Thompson 采样。

两个 prompt 描述的是同一件事。但第二个版本，AI 第一次回复就给出了我能直接用的实现方案。第一个版本，我们来回了四轮才对齐。

**差距不是 AI 的理解能力，而是我给出的信息质量。**

## 意外收获：口述逼你想清楚

这是我最没预料到的变化。

打字的时候，我可以「边写边想」——想到哪写到哪，发出去再说。但说话不行。说话需要连贯，语言结构不自洽的时候你会感觉到，因为你自己先听到了。

有好几次，我对着麦克风刚开口，说到一半就停下来了——因为我发现自己其实还没想清楚。

这是个过滤器。

它把「我以为我想清楚了」和「我真的想清楚了」区分开来。后者才值得发给 AI。

结果是：语音输入之后，我的 prompt 返工率明显下降。在搭建 dingning.ai 的博客系统期间，有几个功能我几乎是「一次描述，一次出结果」，没有经历反复拉锯的过程。不是 AI 变聪明了，是我的输入变清晰了。

## 语音不是万能的：我的取舍原则

用了一段时间，我也摸清了语音输入的边界。

**适合语音的场景：**

- 描述产品逻辑、功能背景、设计意图
- 撰写结构性 prompt（尤其是需要交代上下文的那种）
- 会议记录、思路整理、草稿口述
- 情绪状态不好、手感很差的时候

**不适合语音的场景：**

| 场景 | 原因 |
|------|------|
| 写代码本身 | 变量名、语法符号、缩进——说出来反而慢 |
| 需要精确引用路径或 API 名称 | 口述容易出错，不如直接粘贴 |
| 嘈杂环境 / 不方便开口 | 识别质量下降，或者场合不合适 |
| 极短的指令 | 「帮我加个换行」——打字比开口快 |

我现在的工作流是混合模式：**需求描述和 prompt 撰写用语音，代码操作和精确引用用键盘**。两者不是竞争关系，而是互补。

## 表达方式，才是 Vibe Coding 最被忽视的变量

大家讨论 Vibe Coding 的时候，话题经常集中在：

- 用哪个 AI 模型？
- 用 Cursor 还是 Windsurf？
- 如何写好 system prompt？

但很少有人认真讨论：**你用什么方式把需求传达给 AI？**

输入法不是小事。它决定了你的想法以什么形态进入 AI——是完整的、有上下文的、有逻辑层次的，还是被键盘效率倒逼着压缩成三行字的。

从「异乡人才」的 Python 后端，到 dingning.ai 的 MDX 博客系统，我写过超过 18,000 行 AI 协作生成的代码。这段经历让我越来越相信一件事：**AI 的输出质量天花板，取决于你的输入质量。** 而语音输入，是我目前找到的、最容易提升输入质量的那个改变。

不是因为它高科技，而是因为说话比打字更难偷懒。

---

你现在的 Vibe Coding 工作流里，有没有哪个环节是靠「凑合着描述」在推进的？不妨试着把它改成口述一遍，看看会发生什么。

*工具改变表达，表达改变输出，输出改变结果——这条链比我们以为的更短。*
