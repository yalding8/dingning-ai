## 引言：当“改崩了”成为最大的心魔

在过去的几个月里，我，一个非科班出身的“野生”开发者，用 Vibe Coding 的方式，独立完成了两个生产级项目：我的个人网站 [dingning.ai](https://dingning.ai) 和 AI 驱动的留学生求职平台“异乡人才”。

这两个项目加起来，有超过 **18,000 行代码**。

听起来或许有点“唬人”，但说实话，敲下每一行代码时，我心里都悬着一把剑。不是担心技术有多难——有 AI 在，实现路径总能问出来。我最深的恐惧，其实是所有非程序员（甚至很多程序员）的共同噩梦：

**“万一我这次改代码，把之前好好的功能搞崩了怎么办？”**

这种恐惧，足以让任何热情在深夜的调试中消耗殆尽。Vibe Coding 绝不是“随意编码”，它的精髓，恰恰在于**用 AI 赋予的工具和效率，去构建比传统开发更严谨的工程纪律**。

今天，我想和你分享的，不是“如何用 AI 写代码”，而是我们如何为 Vibe Coding 编织一张**系统性的“安全网”**。这张网由三层构成，它让我这个非程序员，也能有底气去修改一个包含 505 个测试的复杂推荐算法。

## 安全网第一层：Git 与原子提交 —— 每一次“存档”都是后悔药

我的编程之旅始于个人网站。当时，AI 助手帮我生成了漂亮的首页。我兴奋地想要调整一个按钮的颜色。改了几下，预览页面一片空白。冷汗瞬间就下来了：我刚才改了哪里？怎么退回去？

那一刻，我深刻理解了 **“版本控制”不是可选项，而是生命线**。

我立刻让 AI 教我 Git 的基础命令：`git init`, `git add`, `git commit`。但更重要的是，AI 教会了我一个黄金实践：**原子提交（Atomic Commit）**。

**什么是原子提交？**
就是不把“修改导航栏”和“修复登录框bug”这两件不相关的事，混在一次提交里。每一次提交，都应该只做一件明确、独立的小事。

**在我的个人网站项目中，提交记录是这样的：**
- `feat: 添加深色模式基础CSS变量`
- `fix: 修复移动端导航菜单点击区域过小的问题`
- `docs: 更新‘关于我’页面内容`

**这样做的好处是爆炸性的：**
1.  **回滚精准**：如果新加的深色模式有问题，我可以单独回退那一次提交，而不会影响其他任何功能。
2.  **历史清晰**：三个月后回头看，我能像看日记一样，清晰追溯每一个功能是如何一步步长出来的。
3.  **信心倍增**：我知道任何一次错误的修改，都有“一键还原”的后悔药。这让我敢于不断尝试和重构。

在更复杂的“异乡人才”项目中，原子提交的粒度控制得更细。例如，开发 Thompson Sampling 推荐引擎时，我的提交序列是：
1.  实现核心概率选择函数
2.  添加模拟数据生成器
3.  编写算法效果可视化脚本
4.  修复边界条件的一个计算错误

**每一次提交，都是一个稳定的“存档点”**。Git 构成了安全网最坚实的第一层：它保证你不会从悬崖上直接坠落，而是退回到上一个安全的台阶。

## 安全网第二层：单元测试作为“活文档”与回归防护

如果说 Git 是“后悔药”，那么单元测试就是“预防针”和“使用说明书”的结合体。

在“异乡人才”项目中，我们需要一个公平的顾问评价排名算法。简单的“平均分”会让新顾问毫无机会，而“好评率”又容易被刷单。我们选择了**Thompson Sampling**——一个能平衡“探索”（给新顾问曝光机会）和“利用”（推荐高分顾问）的智能算法。

这个算法涉及概率分布（Beta分布）、随机抽样，对我而言如同天书。但 AI 不仅解释了原理，还做了一件至关重要的事：**它引导我，为这个复杂算法编写了 505 个单元测试。**

**这些测试，成为了我理解算法的“活文档”。**
我看不懂数学公式，但我能看懂测试用例：

```python
def test_thompson_sampling_favors_high_performance():
    # 模拟两个顾问：A（好评100，差评20） vs B（好评10，差评2）
    # 算法在大量模拟中，选择A的次数应该显著高于B
    ...
```

每一个测试，都是一个具体的业务场景。通过阅读这 505 个测试，我——一个产品负责人——彻底明白了这个算法在**各种极端情况**下会如何表现：新顾问上线时、出现恶意差评时、样本量很小时……

**更重要的是，它们构成了坚不可摧的回归防护网。**
当我后来需要修改算法的一个参数，或者优化一段性能代码时，我只需要运行：

```bash
pytest
```

如果 505 个测试全部通过（绿色），我就有 **99.9% 的把握**，我的修改没有破坏任何现有逻辑。这种安全感，是任何“人肉检查”都无法给予的。

**Vibe Coding 模式下，编写测试的范式改变了：**
1.  **先写测试，再写实现（TDD）**：我会让 AI “为某个功能先设计测试用例”。这迫使我在写代码前就想清楚边界条件。
2.  **AI 生成测试骨架**：描述功能后，让 AI 生成测试函数框架，我再去填充具体的断言。
3.  **测试即沟通**：我把测试用例作为和 AI 沟通需求的精确语言。“你看，这个测试模拟了用户连续刷到同一个顾问的情况，我们的算法能避免吗？”

这 505 个测试，是项目里最宝贵的资产之一。它们让 18,000 行代码不再是“黑盒”和“屎山”，而是一个**行为可预测、变更可验证**的可靠系统。

## 安全网第三层：Gate-Review 与 AI 模拟专家评审

前两层网，保护的是“代码实现”不出错。但最致命的风险，往往出现在更早的阶段：**产品设计和方案设计阶段就跑偏了。**

一个功能，代码写得再漂亮，测试覆盖率 100%，如果它本身是个错误的需求，那就是在完美地浪费生命。

为此，我借鉴了软件工程的“门禁（Gate）”概念，建立了一套 **Gate-Review 流程**。在投入大量编码时间之前，设置必须通过的评审点。

**我的 Gate-Review 核心是：让 AI 模拟专家团进行挑战。**

例如，在“异乡人才”项目中，设计“岗位匹配度计算模型”时，我没有直接开始写代码。而是先让 AI 进行了一场模拟评审：

**我（作为产品经理）提交方案：**
> “计划用加权求和法：学历权重30%，专业匹配度权重40%，技能关键词权重30%。”

**我指令 AI 扮演的“专家团”进行评审：**
> “请你分别扮演一位数据科学家、一位资深HR、一位留学生用户，从你们的专业角度，挑战这个方案的合理性。”

**AI 返回的“评审意见”摘要：**
- **数据科学家**：“加权求和简单，但线性模型无法捕捉各维度间的交互关系。例如‘顶尖名校+专业略不匹配’ vs ‘普通学校+专业完全匹配’，哪个更好？建议考虑机器学习模型或更复杂的规则引擎。”
- **资深HR**：“权重设置过于静态。对于技术岗，技能权重要远高于学历；对于管培生，学校背景可能更重要。需要分类处理。”
- **留学生用户**：“我关心的是‘真实入职可能性’。有些岗位要求‘精通方言’，这一条对我就是绝对门槛，不是权重高低的问题。”

这场十分钟的模拟评审，**直接否定了我的第一版方案**，避免了至少一周的无效开发。我随后调整方向，设计了一个分岗位类型、结合绝对门槛与弹性评分的混合模型。

**Gate-Review 的关键在于：**
1.  **在编码前设置“门”**：设计稿、技术方案、数据库Schema都必须先通过评审。
2.  **利用 AI 进行多角色挑战**：让它扮演用户、业务方、技术专家、竞争对手，从不同视角“找茬”。
3.  **记录决策依据**：为什么采用A方案而非B？把AI的分析和你的判断记录下来，成为未来的“项目记忆”。

这一层安全网，守护的是**方向正确**。它确保我们是在“做正确的事”，而不仅仅是“把事做对”。

## 总结：Vibe Coding，是纪律的进化，而非纪律的消亡

回顾从 505 个测试到 18,000 行代码的旅程，我想传递的核心信息是：

**Vibe Coding 不是编程纪律的终结，而是它的民主化与智能化演进。**

过去，严格的工程实践（版本控制、测试、评审）是资深工程师用经验和时间构筑的高墙，将非专业人士挡在门外。现在，AI 成为了我们每个人的“资深协作者”，它将这些实践的能力和门槛，前所未有地降低了。

- **Git 和原子提交**，AI 可以在 5 分钟内教会你，并成为你的肌肉记忆。
- **单元测试**，不再是枯燥的负担，而是你理解复杂逻辑的“说明书”和修改代码的“勇气之源”。
- **设计评审**，不再需要苦苦预约专家时间，一个提示词就能发起一场多维度的头脑风暴。

对于每一位想用 Vibe Coding 构建真实产品的管理者、业务人员，我的建议是：

**请像重视你的商业计划书一样，重视你的“工程安全网”。** 它不是你成为程序员后的“选修课”，而是你用 AI 将想法可靠地变为现实的“必修课”。

从今天起，在你的下一个 Vibe Coding 项目中，尝试实践这三层安全网：
1.  初始化一个 Git 仓库，并承诺你的每一次提交都是“原子”的。
2.  为一个核心函数，让 AI 帮你编写 3-5 个涵盖正常和异常情况的测试。
3.  在画完原型图后，让 AI 模拟一次“专家评审会”，挑战你的设计。

你会发现，当恐惧被系统性地消除，创造力才会真正获得自由。

*用 AI 赋予的敏捷，去践行更深刻的严谨——这或许是非技术背景构建者，在这个时代最浪漫的逆袭。*
