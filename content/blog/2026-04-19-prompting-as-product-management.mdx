很多人跟我说，他们试过用 AI 写代码，但效果很差——要么生成的东西驴唇不对马嘴，要么改了三遍还是不对，最后放弃了。

我问他们：你是怎么跟 AI 说的？

「就说了一句话，让它帮我做一个推荐系统。」

问题找到了。

---

## 把 Prompt 当搜索框，是大多数人失败的根源

搜索框的逻辑是：你丢一个关键词，引擎去匹配。

但 AI 不是搜索引擎。它更像一个能力很强、但什么背景都不知道的新员工。你给它一个模糊的任务，它会「尽力完成」——但「尽力」不等于「做你想要的」。

我在做异乡人才项目的时候，早期也犯过这个错。2026 年 2 月项目启动，我对着 Claude Code 输入：

> 「帮我做一个岗位推荐引擎」

它确实生成了代码。但那个推荐逻辑完全不是我想要的——它用的是最简单的关键词匹配，而我需要的是基于学历、专业、语言能力多维度加权的智能匹配。

我花了半天时间才搞清楚：**不是 AI 不够聪明，是我没有把需求说清楚。**

那一刻我意识到，我需要换一种思维方式——不是用户思维，而是**产品经理思维**。

---

## 我的 Prompt 进化史：从一句话到结构化文档

做产品这些年，我写过很多需求文档（PRD）。PRD 的核心不是字数多，而是让任何人读完都能理解：**做什么、为什么做、边界在哪、怎么算做完了**。

我开始把这套逻辑移植到 Prompt 上。

**第一阶段：一句话 Prompt（失败率极高）**

```
帮我做一个岗位推荐引擎
```

结果：AI 自由发挥，方向对但细节全错。

**第二阶段：加了背景和目标（好了一点，但还缺约束）**

```
我在做一个留学生求职平台，需要一个推荐引擎，
根据用户的学历、专业、语言能力匹配合适的岗位。
用 Python + FastAPI 实现。
```

结果：方向对了，但边界不清，AI 加了很多我不需要的功能。

**第三阶段：结构化文档 Prompt（成功率大幅提升）**

我开始用固定的结构写每一个功能需求，像写 PRD 一样。

## 三个真实项目的 Prompt 长什么样

### 案例一：异乡人才推荐引擎

这是我在异乡人才项目里写的实际 Prompt 结构（简化版）：

```
## 背景
异乡人才是一个服务留学生的求职平台，目前有 10,000+ 个活跃岗位。
用户画像：海外留学生，有回国就业意向，学历从本科到博士不等。

## 目标
实现一个岗位推荐引擎，为每个用户返回 Top 20 个最匹配的岗位。

## 匹配维度（按权重排序）
1. 专业方向匹配（权重 40%）
2. 学历层次匹配（权重 30%）
3. 语言能力（权重 20%）
4. 工作经验年限（权重 10%）

## 技术约束
- 后端：Python + FastAPI
- 数据库：PostgreSQL
- 缓存：Redis（推荐结果缓存 1 小时）
- 返回格式：JSON，包含 job_id、score、match_reason

## 验收标准
- 单个用户推荐响应时间 < 200ms
- 推荐结果可解释（每条结果附带匹配理由）
- 有单元测试覆盖核心匹配逻辑

## 不要做的事
- 不要引入机器学习模型（规则引擎即可）
- 不要实现用户行为追踪（这是下一期需求）
```

这个 Prompt 大概 300 字。但它生成的代码，我几乎不需要大改——因为 AI 知道了它不该做什么，和它必须做什么。

### 案例二：异乡点评评分系统

异乡点评是另一个项目，核心是顾问评价系统。早期我用简单平均分排名，但这有一个经典问题：一个只有 2 条评价、都是 5 星的顾问，会排在一个有 100 条评价、平均 4.8 分的顾问前面。这不公平。

我在 Prompt 里是这样描述这个问题的：

```
## 背景
顾问评价系统目前用简单平均分排名，存在冷启动问题：
评价数量少的顾问因为偶然的高分被排到前列，对用户产生误导。

## 目标
引入 Thompson Sampling 算法替代简单平均分，
让排名同时考虑「评分高低」和「评价数量」两个维度。

## 算法说明
Thompson Sampling：把每个顾问的评价建模为 Beta 分布，
正向评价（≥4星）作为成功次数，负向评价作为失败次数，
通过采样 Beta 分布来决定排名，自动平衡探索与利用。

## 验收标准
- 新顾问（0条评价）的初始分布与全体顾问均值一致
- 有 100+ 条评价的顾问，其排名稳定性 > 90%
- 单元测试：至少覆盖冷启动、高评分少量评价、低评分大量评价三个场景

## 反例
不要用贝叶斯平均（Bayesian Average），
那个方法需要预设先验均值，不适合我们的冷启动场景。
```

这个 Prompt 做了一件很重要的事：它解释了**为什么**用这个算法，还明确说了**不要用哪个算法**。结果是跨项目累计写出了 505 个单元测试，推荐算法的可靠性有了真实保障。

### 案例三：Gate-Review 流程

到了 dingning.ai 个人网站项目（2026 年 3 月上线），我已经形成了一套固定的开发流程：每个功能上线前，必须经过 Gate-Review——代码质量检查、测试覆盖、UI 一致性三道门。

为了让 AI 帮我执行这个流程，我写了一个「流程型 Prompt」：

```
## 背景
dingning.ai 是一个用 Next.js 14 + MDX 搭建的个人博客网站，
设计系统基于 CSS 变量，主色调为暖白底色 + 海军蓝强调色。

## Gate-Review 检查任务
在我提交 PR 之前，请按以下三道门检查当前代码变更：

### Gate 1：代码质量
- TypeScript 无类型错误
- 无 console.log 遗留
- 组件命名符合 PascalCase

### Gate 2：测试覆盖
- 新增功能有对应测试
- 现有测试全部通过

### Gate 3：UI 一致性
- 新增 UI 使用 CSS 变量而非硬编码颜色
- 中文文案，无英文混入导航和标签

## 输出格式
请逐条列出检查结果，通过用 ✅，未通过用 ❌ 并说明原因。
```

这个 Prompt 不是让 AI 写代码，而是让它扮演「质检员」。它奏效了——因为我把验收标准写得足够具体，AI 能对着标准做判断，而不是凭感觉。

## 好 Prompt 的五要素

把三个项目的经验提炼一下，我认为一个有效的 Prompt 需要包含：

| 要素 | 作用 | 常见缺失后果 |
|------|------|------------|
| **背景** | 告诉 AI 这是什么产品、谁在用 | AI 做出通用解而非定制解 |
| **目标** | 明确本次任务要达成什么 | AI 做了很多，但没做对的 |
| **约束** | 技术栈、性能要求、不能做什么 | AI 自由发挥，引入不需要的依赖 |
| **验收标准** | 怎么算做完了、怎么算做对了 | 结果无法判断好坏，反复返工 |
| **反例** | 明确哪些路径不要走 | AI 选了你不想要的方案 |

五要素不是每次都需要全写，短任务可以省略反例；但**背景、目标、验收标准**这三项，我几乎从不省略。

## AI 时代最值钱的能力，不是写代码，是写需求

我不是程序员。我用 Vibe Coding 的方式，在异乡人才项目里写出了 18,000+ 行代码；在 dingning.ai 里搭起了完整的博客系统和 CI/CD 流程。

但让我真正能做到这些的，不是我学会了 Python 或 TypeScript，而是我**学会了如何把一个模糊的想法，转化成一份 AI 能理解的需求文档**。

这件事，产品经理、运营、设计师都能做。甚至比很多工程师做得更好——因为他们天然懂得从用户视角描述问题，而不是从技术视角描述解法。

如果你也想开始用 AI 做自己的产品，我建议你在打第一个字之前，先问自己三个问题：

1. 我要解决的是什么问题？
2. 做完之后，我怎么知道它做对了？
3. 有什么是我明确不想要的？

把这三个问题的答案写下来，你的 Prompt 就已经比 80% 的人强了。

---

*会写需求的人，才是 AI 时代真正的「10x 开发者」——哪怕他们一行代码都不会写。*
