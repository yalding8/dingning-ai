三周前，我在 dingning.ai 上线了 Newsletter 订阅功能。

那一刻我盯着屏幕上那个简陋的邮箱输入框，心里同时装着两种感觉：一种是「终于做成了」的松了口气，另一种是「现在怎么办」的茫然。

技术上跑通只是开始。真正难的，是从零开始让人愿意把邮箱交给你。

---

## 为什么不用公众号？

这个问题，身边几乎每个人都问过我。

公众号的逻辑我当然懂——流量大、发现机制完善、用户习惯已经养成。但我做这个网站，本来就不是为了追流量。

我更在意的，是几个具体的问题：

- **我的读者能不能在算法之外找到我的内容？** 公众号的分发逻辑是平台决定的，不是我决定的。
- **如果某天平台规则变了，我的内容还在吗？** 我在公众号写的东西，理论上不属于我。
- **我能不能直接触达每一个真正关心我在做什么的人？** 邮件能，公众号推送不稳定。

我希望订阅者和我之间，是一条我能控制的直线，而不是一条随时可能被掐断的平台通道。

所以选择了 Buttondown。它的定位就是「给独立写作者用的 Newsletter 工具」，数据是我的，内容是我的，订阅者列表也是我的。

这是一个**内容自主权**的选择，不是技术偏好。

---

## 用 Vibe Coding 把 Buttondown 接进 Next.js

我不是程序员。这件事我在上一篇文章里说过了，这里不重复。

但我想专门讲一下这次集成的过程，因为它比我想象的更曲折——也更有趣。

### 第一步：搞清楚 Buttondown 的 API 逻辑

Buttondown 提供一个非常简单的 REST API，核心只有一个动作：把邮箱地址 POST 到他们的订阅端点。

理论上，十行代码就能搞定。

但我犯的第一个错误，是直接在前端组件里写 API Key。Claude 提醒我这样做会把密钥暴露在浏览器里，安全风险很大。这是我之前完全没意识到的事。

所以第一个坑是：**前端不能直接调用第三方 API，必须走自己的后端路由。**

### 第二步：写 Next.js 的 API Route

Next.js 14 的 App Router 支持在 `app/api/` 目录下写服务端路由。我让 Claude 帮我生成了这段逻辑：

```
POST /api/subscribe
→ 接收前端传来的 email
→ 在服务端用 BUTTONDOWN_API_KEY 调用 Buttondown API
→ 返回成功或错误信息给前端
```

这部分本身不复杂。但第二个坑来了：**环境变量的配置。**

我在本地 `.env.local` 文件里配好了 API Key，本地测试完全正常。部署到 Vercel 之后，订阅功能直接报错。

折腾了将近一个小时，才发现问题：Vercel 的环境变量需要在项目设置里单独配置，不会自动读取本地 `.env` 文件。配好之后，重新部署，一切正常。

### 第三步：前端交互的打磨

功能跑通之后，我花了更多时间在用户体验上：

- 提交后显示「✓ 已订阅，感谢！」而不是页面跳转
- 重复订阅时返回友好提示，不报错
- 加了简单的 loading 状态，防止用户重复点击

这些细节，是 Claude 在我描述「用户会怎么操作」之后主动建议的。Vibe Coding 的精髓在这里——**你描述体验，AI 翻译成代码。**

整个集成过程，从开始到上线，大概用了半天。坑是真实的，但都是可以解决的坑。

## 冷启动的 0 → 1：第一批订阅者从哪来

技术跑通的那天，订阅人数是 0。

我没有做任何付费推广，也没有靠转发裂变。第一批订阅者，来源非常具体：

**1. 博客文章末尾的订阅入口**

每篇文章底部，我都放了一个订阅模块，文案是：「如果这篇文章对你有用，可以订阅我的 Newsletter，我会把没有公开发布的思考直接发到你的邮箱。」

这句话的关键词是「没有公开发布的」——给订阅行为一个明确的理由。

**2. 向已有的真实读者直接告知**

网站上线后，我主动联系了几位我知道在读我内容的朋友，告诉他们 Newsletter 上线了，直接给了他们链接。不是群发，是一对一的信息。

这批人转化率很高，因为他们本来就关心我在做什么。

**3. 社交媒体上的轻量曝光**

我在 LinkedIn 上发了一条帖子，说「Newsletter 开始了，第一期聊 XX 话题」，附上了订阅链接。没有强调「快来订阅」，而是把内容本身放在前面。

## Newsletter 与博客的内容分工

这是我花时间最长想清楚的一件事。

如果 Newsletter 发的内容和博客一样，人们为什么要多一个渠道？如果 Newsletter 是博客的摘要，又显得太随意。

我最终的分工逻辑是这样的：

| 维度 | 博客文章 | Newsletter |
|------|----------|------------|
| 内容类型 | 系统梳理、方法论、完整案例 | 过程记录、未成形的想法、当周观察 |
| 发布节奏 | 不定期，质量优先 | 每周一封，保持节奏 |
| 目标读者 | 搜索流量 + 新访客 | 真正关注我动向的人 |
| 写作状态 | 打磨过的 | 相对原始、直接 |

简单说：**博客是我说给所有人听的，Newsletter 是我说给真正关心的人听的。**

这个定位让我在写 Newsletter 的时候轻松很多——我不用追求完美，我只需要真实。

---

## 运营三周的真实数据复盘

说几个真实数字，以及我从中得到的意外收获。

**打开率**

前三封邮件的打开率分别在 60% 左右。这个数字比我预期的高很多。行业平均水平大概在 20-30%，早期小规模名单的打开率通常更高，但 60% 还是让我有点意外。

原因可能是：早期订阅者都是真正认识我、关心我在做什么的人，而不是随机流量。

**转化路径**

大部分订阅者来自博客文章末尾的入口，而不是社交媒体的直接链接。这说明「内容本身」是最好的转化入口——读到一半觉得有用的人，比看到一条推广帖的人，转化意愿高得多。

**最出乎意料的发现**

有人在收到第一封 Newsletter 之后，回复了我。

不是一个字的回复，是一段有内容的回复，聊到了他自己在做的类似探索。

这件事让我意识到，Newsletter 和博客最大的区别不是技术层面的，而是**关系层面的**——邮件是一种对话，博客更像是一场演讲。订阅者回复你，是因为他们觉得这封信是写给他们的，不是发给所有人的。

这个感受，是公众号给不了我的。

---

## 如果你也想做 Newsletter

几个最实用的建议，来自这三周的真实经历：

1. **先想清楚为什么，再去选工具。** 工具本身不重要，你想和读者建立什么关系才重要。
2. **不要等「准备好了」再启动。** 我第一封邮件的订阅人数是个位数，但我还是发了，因为等待不会让人数变多。
3. **内容分工要明确。** 如果 Newsletter 是博客的复制粘贴，人们不会订阅。
4. **Buttondown 对非技术人友好。** API 逻辑简单，文档清晰，Vibe Coding + 半天时间可以搞定集成。

---

我还在这个早期阶段摸索，数据量不大，结论也是暂时的。但有一件事我已经想清楚了：**我不需要很多订阅者，我需要真正在乎我在做什么的那些人。**

这是一条慢的路，但是我自己的路。

*私域的本质不是圈人，是找到真正彼此在乎的人。*
