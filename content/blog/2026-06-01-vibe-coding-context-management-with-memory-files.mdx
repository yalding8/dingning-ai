如果你也在用 AI 写代码，你一定遇到过这种情况：

新开一个对话，AI 开始帮你干活，一切看起来都很顺。但没多久，你发现它把 A 项目的数据说成了 B 项目的，或者把你的职位写错了，或者声称你们公司用了某个你们根本没用过的工具。

你纠正它，它道歉，下一条继续错。

这不是 AI 变笨了。这是 **AI 的记忆是会话级的**——每次新对话，它对你的项目一无所知。

---

## AI 的「失忆」，是 Vibe Coding 最被忽视的工程问题

大多数关于 Vibe Coding 的文章，讲的是「怎么提示词写得更好」「怎么让 AI 帮你调 Bug」。很少有人认真讨论：**当你同时在跑三个项目，AI 怎么知道它现在在哪个项目里？**

我目前在并行维护三个项目：

- **异乡人才**（uhomes-talent）：AI 驱动的留学生求职平台，2026 年 2 月启动，Python + FastAPI + React
- **异乡点评**（dianping）：NestJS + Nuxt，带 505 个单元测试
- **dingning.ai**：我的个人网站，2026 年 3 月上线，Next.js 14 + MDX

三个项目技术栈不同，数据不同，背景不同。但 AI 不知道这些。

每次新开对话，它都是一张白纸。

---

## 真实翻车现场：AI 如何把我的项目搞乱

我来说两个真实发生过的错误，不是假设，是实际踩过的坑。

**错误一：指标归属混乱**

我让 AI 帮我写一段关于异乡人才的项目介绍，里面提到了单元测试覆盖率。AI 很自然地写道：「异乡人才目前已有 505 个单元测试……」

问题是，**505 个单元测试属于异乡点评，不是异乡人才。**

两个项目在同一个对话里聊过，AI 把指标混淆了。如果我不仔细核对，这条错误信息就会出现在博客、产品页，甚至对外的 PR 稿里。

**错误二：称谓错误**

写公司介绍时，AI 把我的身份写成了「创始人」。

我在异乡好居的身份是**合伙人**，不是创始人。这两个词在中文商业语境里有很大的区别。但对 AI 来说，它只是在「合理推断」——你在写这家公司，你在操盘，叫你创始人好像也没什么问题。

它不知道，但我知道。

---

## 我的解法：用「记忆文件」给 AI 一个项目大脑

从那以后，我开始为每个项目维护一份结构化的 **memory 文件**。

这不是什么复杂技术，就是一个 Markdown 文档。但它是我和 AI 协作的「事实基准」——**只要我在对话开始时把这个文件贴给 AI，它就不会再犯上面那两类错误。**

我目前维护了三份 memory 文件：

| 文件名 | 用途 |
|--------|------|
| `company-facts.md` | 公司级事实清单，覆盖工具、数据、称谓 |
| `uhomes-talent.md` | 异乡人才项目的技术栈、指标、时间线 |
| `dingning-ai.md` | 个人网站的技术决策、开发历程、关键约束 |

### memory 文件里写什么？

以我的 `company-facts.md` 为例，它包含以下几个核心模块：

**① 公司使用的工具（已确认）**

明确列出哪些工具在用、哪些在关注但未启用、哪些明确未使用。

比如：飞书是「在关注，未启用」——AI 可以在文章里推荐飞书，但不能说「我们用飞书」。这条规则如果不写进 memory，AI 根本不知道。

**② 业务数据与归属**

| 数据 | 值 | 归属 |
|------|-----|------|
| 505 个单元测试 | — | **异乡点评** |
| 18,000+ 行代码 | — | 异乡人才 |
| 10,645 个活跃岗位 | — | 异乡人才 |

每条数据都标注归属项目，杜绝跨项目混淆。

**③ 称谓规范**

- Ning Ding 在异乡好居的角色：**合伙人**
- dingning.ai 角色：**主理人**

两个字的差异，背后是完全不同的含义。写进去，AI 就不会乱推断。

**④ 技术栈事实**

| 项目 | 后端 | 前端 | 数据库 |
|------|------|------|--------|
| 异乡人才 | Python + FastAPI | React | PostgreSQL + Redis |
| 异乡点评 | NestJS | Nuxt | PostgreSQL + Redis |
| dingning.ai | — | Next.js 14 | 无（静态 MDX） |

两个项目都用了 PostgreSQL + Redis，但后端框架完全不同。这种「相似但不同」的信息，最容易被 AI 搅混。

## memory 文件的四条设计原则

用了几个月之后，我总结出这套方法的核心原则：

**① 最小化**

只写 AI 有可能搞错的事实，不要把项目文档全塞进去。越精简，AI 越能抓住重点。

**② 可验证**

每条信息都标注「用户确认日期」。不是我记得，是我确认过的。这让我自己也有追溯的依据——如果某条数据更新了，我知道要去改这份文件。

**③ 分项目维护**

公司级事实放 `company-facts.md`，项目级事实放各自的文件。不要把所有东西塞进一个文档，那样找起来麻烦，也容易让 AI 混淆边界。

**④ AI 不得自行添加**

这是最重要的一条规则：**memory 文件只由我确认后更新，AI 不能主动往里加内容。**

AI 很聪明，它会「帮你补充」。但它补充的可能是推断，不是事实。memory 文件的价值，恰恰在于它只包含已确认的真相。

---

## 这套方法不只适用于写代码

我最初设计 memory 文件是为了 Vibe Coding，但很快发现它在其他场景同样有效：

**写作助手**：我用 AI 写博客时，`company-facts.md` 是每次对话的开场贴附。它保证文章里的公司介绍、数据引用不会出错。本篇文章就是在 memory 文件约束下生成的。

**内容审核**：让 AI 帮你检查一篇文章是否有事实错误，前提是它得知道什么是事实。把 memory 文件给它，它才能判断「这句话和我们实际使用的工具对不上」。

**产品文档生成**：技术栈、接口设计、关键决策——这些东西如果 AI 不知道，它生成的文档要么太泛，要么出错。memory 文件让它生成的文档有据可查。

本质上，**memory 文件解决的是「AI 如何知道你的现实」这个问题**。它适用于任何你想让 AI 长期、准确地帮你处理的场景。

---

## 操作层面：怎么开始？

不需要一次性把所有东西都整理好。我的建议是：

1. **从一次翻车开始**：下次 AI 说错了什么，把那条错误信息的「正确版本」写进 memory 文件
2. **每个项目单独一个文件**：用 Markdown，放在项目根目录，命名为 `memory.md` 或 `project-facts.md`
3. **对话开始时贴进去**：不需要复杂的工具，直接复制粘贴到对话框里，告诉 AI「这是本项目的事实基准，请严格遵守」
4. **只在你确认后更新**：AI 如果提议修改 memory，先核实，再决定

这件事的门槛非常低。你不需要会写代码，不需要懂工程架构，只需要一个文本编辑器和一点点整理信息的习惯。

---

## 最后

Vibe Coding 让非程序员能以前所未有的速度构建产品。但速度快，出错也快。

**AI 的失忆不是它的问题，是我们没给它足够的记忆。**

memory 文件不是在限制 AI，而是在给它一个可以信赖的现实。当 AI 知道你是谁、你的项目是什么、你的数据属于哪里，它才能真正成为你的协作者，而不是一个每次都要重新认识你的陌生人。

如果你也在做多项目并行的 Vibe Coding，我很好奇：你现在是怎么管理 AI 的上下文的？

*上下文管理，是 Vibe Coding 里最不性感、却最值得投入的工程细节。*
