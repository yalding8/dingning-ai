# dingning.ai 项目规则

## 博客 Slug 规则（重要）

Vercel SSG 静态部署不支持非 ASCII 文件名路由（中文 URL percent-encoded 后无法匹配中文文件名的静态 HTML）。

**必须遵守：**
- 博客文件名和 slug 只允许 ASCII 字符（英文字母、数字、连字符）
- 文件命名格式：`YYYY-MM-DD-english-slug.mdx`，禁止中文出现在文件名中
- 如果文章标题是中文，slug 使用英文翻译/概括，例如标题「18000 行代码」→ slug `18000-lines-vibe-coding-real-cost`
- frontmatter 中可选 `slug` 字段覆盖文件名推导的 slug，优先级高于文件名
- `scripts/generate_blog_post.py` 生成 slug 时必须过滤掉所有非 ASCII 字符

**反例（禁止）：**
```
2026-03-09-18000-行代码我从没叫过一个程序员.mdx  ← 会导致 Vercel 404
```

**正例：**
```
2026-03-09-18000-lines-vibe-coding-real-cost.mdx  ← 正确
```

## 内容一致性规则（重要）

所有博客文章、播客等内容发布前必须通过一致性检查，不通过不允许合并上线。

**核心原则：同一事实在所有文章中必须前后一致，不能自相矛盾。**

### 检查维度
1. **数字一致性**：代码行数、测试数量、岗位数量等指标在不同文章中不能冲突（合理增长除外）
2. **属性一致性**：人物年龄、职位、角色描述不能前后矛盾
3. **时间线自洽**：事件发生的时间顺序不能冲突
4. **技术栈一致性**：使用的技术、工具描述不能矛盾（一篇说 PostgreSQL，另一篇不能说 MySQL）

### 检查流程
- **CI 门禁**：PR 包含博客内容变更时，`Content Consistency Check` job 自动运行
- **自动博客**：`daily-blog.yml` 生成文章后、创建 PR 前自动检查
- **双层检查**：第一层正则规则（快速免费）+ 第二层 LLM 语义分析（深层）
- **阻塞合并**：检查不通过时 CI 失败，PR 无法合并

### 写作时注意
- 引用项目数据前，先查看 `memories/` 目录下的最新数据
- 数字如有更新，应在文中说明是最新数据（如"截至 2026 年 3 月"）
- 不确定的数字用"约"、"近"等模糊表达，避免与其他文章冲突

### 检查脚本
- `scripts/check_content_consistency.py`
- 手动运行：`python scripts/check_content_consistency.py --all`
- 检查单篇：`python scripts/check_content_consistency.py --target content/blog/xxx.mdx`

## CI/CD 注意事项

- **GITHUB_TOKEN 限制**：由 `peter-evans/create-pull-request` 使用 `GITHUB_TOKEN` 创建的 PR 不会触发 `code-quality.yml`（GitHub 安全策略防循环）
- **自动博客的质量检查**：已内置在 `daily-blog.yml` 中（一致性检查 + build 验证），不依赖 Code Quality workflow
- `code-quality.yml` 仅覆盖人工创建的 PR
- 如需自动博客 PR 也触发 Code Quality，需要用 PAT 替代 GITHUB_TOKEN

## 内容真实性规则（重要 — 品牌安全）

**核心原则：dingning.ai 是个人品牌网站，所有内容必须真实可溯源。虚构内容会直接伤害品牌信誉。**

### 绝对禁止

1. **禁止编造数据**：不得凭空生成业务指标（客户数、交易额、合作伙伴数等）。所有数字必须来自用户提供的事实或可验证的代码仓库
2. **禁止混淆项目归属**：不同项目的指标不得混在一起描述。例如：
   - 异乡人才的代码行数 ≠ 异乡点评的测试数量，不能合并说"18000 行代码 + 505 个测试"
   - 必须明确标注哪个数字属于哪个项目
3. **禁止编造市场数据**：行业规模、市场份额、竞品数据等，如无可靠来源，不得写入文章。使用时必须标注来源或用模糊表达（"据行业估算"）
4. **禁止编造案例/故事**：博客中的案例必须基于真实事件，不得为了"讲好故事"而虚构细节
5. **禁止以真人名义做虚构背书**：不得使用真实公众人物的名字暗示他们参与了评审、推荐或背书（AI 模拟评审应使用虚构角色或明确标注"AI 模拟"）

### 数据分级

| 级别 | 来源 | 使用规则 |
|------|------|---------|
| A 级：代码可验证 | GitHub 仓库、CI 产出、自动统计 | 可直接引用，注明"截至 YYYY-MM" |
| B 级：用户提供 | 用户在对话中确认的公司数据 | 可引用，写入 memories/ 存档 |
| C 级：第三方公开数据 | 行业报告、官网公布 | 需标注来源 |
| D 级：AI 推断/估算 | AI 根据上下文推算的数据 | **禁止作为事实引用**，只能用"约"、"据估算"等模糊表达 |

### memories/ 目录维护规则

- memories/ 是博客生成的事实素材源，其中的错误会被放大到所有未来文章
- 修改 memories/ 文件时，必须确保技术栈、数字、时间线与实际代码仓库一致
- 每次发现 memories/ 中的错误，立即修正并在 git commit message 中说明原因
- 自动同步脚本（sync_memories.py）的产出也需人工审核

### "合理推测"陷阱（重点防范）

AI 最危险的虚构不是明显的错误，而是"看起来很合理的推测"。例如：
- 看到"管理 3 万合作伙伴"→ 推测一定用了飞书 → 编造整篇飞书使用体验 ❌
- 看到"留学缴费"→ 推测市场规模 → 编造"2000 亿"这个数字 ❌
- 看到"非程序员写代码"→ 推测一定从零开始 → 编造"咖啡馆里打开空白编辑器"的故事 ❌

**防范规则：**
1. 任何声称"公司使用 X 工具/流程"的内容，必须在 `memories/company-facts.md` 中有记录。未记录的，不得声称
2. 任何涉及公司内部做法的描述（工具选型、团队流程、管理方式），必须来自用户明确提供的信息，不得由 AI 推断
3. 推荐类文章如果作者没有亲身使用经验，必须明确标注信息来源（"朋友推荐"/"行业口碑"），不得伪装成第一人称使用体验

### 博客生成安全规则

- `generate_blog_post.py` 生成的文章必须经过人工审核后才能发布
- AI 生成的行业数据、市场分析、竞品对比部分应标记为需要人工验证
- 生成后应检查：是否有编造的数字？是否有张冠李戴的指标？是否有虚构的用户故事？
- **新增检查项**：是否声称公司使用了某工具但 `memories/company-facts.md` 中无记录？

### 内容审计记录

- 全站内容审计报告：`docs/AUDIT_2026-03-15_CONTENT_INTEGRITY.md`
- 发现问题后更新报告状态标记，确保所有问题可追溯

## 技术栈
- Next.js 14 + Tailwind CSS v3 + MDX
- 部署：Vercel（SSG 静态生成）
- 博客内容：`content/blog/` 目录下的 `.mdx` 文件
- slug 逻辑：`lib/mdx.ts`（优先 frontmatter slug，否则从文件名去日期前缀）
